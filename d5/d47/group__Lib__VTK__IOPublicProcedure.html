interface Lib_VTK_IO::VTK_GEO

Function for saving mesh with different topologies in VTK-legacy standard.

VTK_GEO is an interface to 8 different functions, there are 2 functions for each of 4 different topologies actually supported: one function for mesh coordinates with R8P precision and one for mesh coordinates with R4P precision.

Remarks:
This function must be called after VTK_INI. It saves the mesh geometry. The inputs that must be passed change depending on the topologies chosen. Not all VTK topologies have been implemented (polydata topologies are absent).
Note:
Examples of usage are:
Structured points calling: 
 ...
 integer(I4P):: Nx,Ny,Nz
 real(I8P)::    X0,Y0,Z0,Dx,Dy,Dz
 ...
 E_IO=VTK_GEO(Nx,Ny,Nz,X0,Y0,Z0,Dx,Dy,Dz)
 ...
Structured grid calling: 
 ...
 integer(I4P):: Nx,Ny,Nz,Nnodes
 real(R8P)::    X(1:Nnodes),Y(1:Nnodes),Z(1:Nnodes)
 ...
 E_IO=VTK_GEO(Nx,Ny,Nz,Nnodes,X,Y,Z)
 ...
Rectilinear grid calling: 
 ...
 integer(I4P):: Nx,Ny,Nz
 real(R8P)::    X(1:Nx),Y(1:Ny),Z(1:Nz)
 ...
 E_IO=VTK_GEO(Nx,Ny,Nz,X,Y,Z)
 ...
Unstructured grid calling: 
 ...
 integer(I4P):: NN
 real(R4P)::    X(1:NN),Y(1:NN),Z(1:NN)
 ...
 E_IO=VTK_GEO(NN,X,Y,Z)
 ...

interface Lib_VTK_IO::VTK_VAR

Function for saving data variable(s) in VTK-legacy standard.

VTK_VAR is an interface to 8 different functions, there are 3 functions for scalar variables, 3 functions for vectorial variables and 2 functions texture variables: scalar and vectorial data can be R8P, R4P and I4P data while texture variables can be only R8P or R4P. This function saves the data variables related to geometric mesh.

Remarks:
The inputs that must be passed change depending on the data variables type.
Note:
Examples of usage are:
Scalar data calling: 
 ...
 integer(I4P):: NN
 real(R4P)::    var(1:NN)
 ...
 E_IO=VTK_VAR(NN,'Sca',var)
 ...
Vectorial data calling: 
 ...
 integer(I4P):: NN
 real(R4P)::    varX(1:NN),varY(1:NN),varZ(1:NN)
 ...
 E_IO=VTK_VAR('vect',NN,'Vec',varX,varY,varZ)
 ...

interface Lib_VTK_IO::VTK_INI_XML

Function for initializing VTK-XML file.

VTK_INI_XML is an interface to 2 different functions, the first used for "common" mesh dimension, where the number of nodes can be represented with I4P integer, while the second used for huge mesh where the number of nodes needs I8P integer representation. In order to use the huge-dimension function an auxiliary variable (logical) must be pass as argument. The XML standard is more powerful than legacy one. It is more flexible and free but on the other hand is more (but not so more using a library like Lib_VTK_IO...) complex than legacy standard. The output of XML functions is a well-formated XML file at least for the ascii format (in the binary format Lib_VTK_IO uses raw-data format that does not produce a well formated XML file). Note that the XML functions have the same name of legacy functions with the suffix XML.

Remarks:
This function must be the first to be called.
Note:
An example of usage is: 
 ...
 integer(I4P):: nx1,nx2,ny1,ny2,nz1,nz2
 integer(I8P):: Hnx1,Hnx2,Hny1,Hny2,Hnz1,Hnz2
 ...
 ! for common dimension mesh
 E_IO = VTK_INI_XML('BINARY','XML_RECT_BINARY.vtr','RectilinearGrid',nx1=nx1,nx2=nx2,ny1=ny1,ny2=ny2,nz1=nz1,nz2=nz2)
 ! for huge dimension mesh (the first logical flag can be either .true. or .false. there is no difference but must be
 ! passed in order to select huge-dimension function)
 E_IO = VTK_INI_XML(.true.,'BINARY','XML_RECT_BINARY.vtr','RectilinearGrid',Hnx1,Hnx2,Hny1,Hny2,Hnz1,Hnz2)
 ...
Note that the file extension is necessary in the file name. The XML standard has different extensions for each different topologies (e.g. vtr for rectilinear topology). See the VTK-standard file for more information.
Returns:
E_IO: integer(I4P) error flag ,

Function for saving mesh with different topologies in VTK-XML standard.

VTK_GEO_XML is an interface to 14 different functions, there are 4 functions for each of 3 topologies supported and a function for closing XML pieces: two functions for mesh coordinates with R8P precision and two for mesh coordinates with R4P precision. There is one function for common dimension mesh (with number of nodes that can be represented by means integer I4P) and another for huge dimension mesh (with number of nodes that needs integer I8P representation).

Remarks:

Note:
